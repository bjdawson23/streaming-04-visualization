## Custom Project

### Dataset

The Dawson pipeline uses the original sales dataset in [data/sales.csv](data/sales.csv).

It contains one row per sales event with fields such as `order_id`, `datetime`, `region_id`, `currency_code`, `product_id`, `unit_price`, `quantity`, `is_online`, `customer_id`, `is_new_customer`, `device_type`, `payment_method`, `referral_source`, `discount_code`, and `customer_note`.

I used the original sales dataset without changing the source records.

### Kafka Messages

The Dawson producer sends sales records from [data/sales.csv](data/sales.csv) to the Kafka topic configured in the project environment.

It uses `region_id` as the Kafka message key so messages from the same region stay ordered together in the same partition.

I kept the message fields the same as the case example, but the Dawson producer defaults to sending 20 messages per run.

### Consumer Processing

The Dawson consumer receives sales messages from Kafka, validates each message, enriches valid records with subtotal, tax, and total fields, and updates a live chart as each message arrives.

It consumes up to 20 messages by default, logs validation results and running totals, and writes accepted records to `data/output/consumed_sales.csv`.

Rejected records are skipped and the valid output fields are written in the project’s standard consumed field order.

### Experiments

The Dawson version includes three small technical changes:

- a dedicated producer module with a default count of 20 messages
- a dedicated consumer module that matches the Dawson naming
- a Dawson live visualization module with custom colors and styling

These changes were small enough to preserve the original workflow while making the project clearly Dawson-specific.

### Results

When I ran the Dawson producer and consumer, the producer published 20 sales messages and the consumer processed them in order, updating the chart as each message arrived.

The live chart saved to `data/output/sales_chart_dawson.png`, and the consumer wrote the accepted rows to `data/output/consumed_sales.csv`.

### Interpretation

This workflow showed how a producer, Kafka topic, consumer, and live visualization fit together as one streaming pipeline.

Compared with the original example, the Dawson version adds custom module names, a 20-message default, and a chart palette that is easier to recognize in the saved output.

For a business, the stream would show sales activity as it happens, making it easier to spot trends in sale totals, message volume, and behavior by region.
